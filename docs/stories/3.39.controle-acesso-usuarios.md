# Story 3.39 — Controle de Acesso por Usuário

## Status: Done

## Contexto

O sistema atualmente trata todos os usuários como administradores. Precisamos de um modelo de permissões que restrinja acesso por **nível de perfil**, por **tela** e por **escopo de dados** (contexto e cliente).

A tela de Gestão de Usuários (Story 3.35) já existe e faz CRUD. Esta story adiciona o modelo de permissões real.

---

## Story

**Como** administrador do sistema,
**Quero** controlar o que cada usuário pode ver e fazer,
**Para** garantir que operadores só acessem os dados e telas autorizados para eles.

---

## Acceptance Criteria

### AC1 — Perfis de acesso (roles)
- [x] Existem pelo menos 3 perfis: `admin`, `operador`, `visualizador`
- [x] `admin`: acesso total a todas as telas e todos os dados
- [x] `operador`: acesso a importação, análise, cálculos e relatórios; sem acesso a configurações de sistema
- [x] `visualizador`: acesso somente leitura a dashboard, relatórios e análise

### AC2 — Controle por tela (módulo)
- [x] Backend valida o perfil do usuário antes de responder endpoints protegidos
- [x] Frontend oculta itens de menu que o usuário não tem acesso
- [x] Tentativa de acesso direto via URL retorna 403 com mensagem adequada

**Matriz de permissões por módulo:**

| Módulo | admin | operador | visualizador |
|--------|-------|----------|--------------|
| Dashboard | ✅ | ✅ | ✅ (leitura) |
| Importar | ✅ | ✅ | ❌ |
| Análise | ✅ | ✅ | ✅ (leitura) |
| Cálculos | ✅ | ✅ | ❌ |
| Relatórios | ✅ | ✅ | ✅ (leitura) |
| Gestão | ✅ | ❌ | ❌ |
| Configurações | ✅ | ❌ | ❌ |

### AC3 — Escopo por contexto (adquirente)
- [x] Usuário pode ser associado a um ou mais contextos (ex: só Cielo, só Rede)
- [ ] Quando o escopo de contexto está definido, o usuário só vê dados desses contextos *(fase 2 — filtros nas queries)*
- [x] `admin` sem restrição de contexto vê todos os contextos
- [x] Tela de gestão de usuário exibe e permite editar os contextos associados

### AC4 — Escopo por cliente
- [x] Usuário pode ser associado a um ou mais clientes específicos
- [ ] Quando o escopo de cliente está definido, o usuário só vê dados desses clientes *(fase 2 — filtros nas queries)*
- [x] `admin` sem restrição de cliente vê todos os clientes
- [x] Tela de gestão de usuário exibe e permite editar os clientes associados

### AC5 — Tela de gestão de permissões
- [x] Na tela de Gestão de Usuários (`/configuracoes/usuarios`), ao editar um usuário:
  - Campo: **Perfil** (dropdown: admin / operador / visualizador)
  - Campo: **Contextos permitidos** (multiselect; vazio = todos)
  - Campo: **Clientes permitidos** (multiselect; vazio = todos)
- [x] Salvar persiste no banco e aplica imediatamente

### AC6 — Banco de dados
- [x] Tabela `usuario_permissoes` com: `usuario_id` (FK → `usuarios.id` ON DELETE CASCADE), `perfil`, `criado_em`, `atualizado_em`
- [x] Tabela `usuario_contextos` com: `usuario_id` (FK → `usuarios.id` ON DELETE CASCADE), `contexto_id` (FK → `contextos.id` ON DELETE CASCADE)
- [x] Tabela `usuario_clientes` com: `usuario_id` (FK → `usuarios.id` ON DELETE CASCADE), `cliente_id` (FK → `clientes.id` ON DELETE CASCADE)
- [x] Migrações criadas e aplicadas (0002_add_permissoes_tables.py)
- [x] Migração inclui `INSERT INTO usuario_permissoes (usuario_id, perfil) SELECT id, 'admin' FROM usuarios` para usuários existentes (sem regressão)
- [x] Rollback: migração `downgrade` remove as 3 tabelas sem afetar dados de `usuarios`

### AC7 — Testes
- [x] Testes unitários para validação de perfil nos endpoints
- [x] Teste: operador não acessa `/configuracoes`
- [ ] Teste: usuário com contexto restrito não vê dados de outro contexto *(fase 2)*
- [ ] Teste: usuário com cliente restrito não vê dados de outro cliente *(fase 2)*
