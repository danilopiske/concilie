## Story

**As a** desenvolvedor do sistema Financial,
**I want** garantir que todas as funcionalidades rodem igualmente em MySQL (desenvolvimento/servidor) e SQLite (distribuição para clientes),
**so that** o sistema possa ser distribuído como aplicativo desktop completo sem perda de funcionalidade, e que upgrades futuros de dependências não quebrem a compatibilidade silenciosamente.

## Acceptance Criteria

1. **AC1 — Substituição do `Session.bind` depreciado**
   - `analista_repository.py` e `calculo_repository.py` não utilizam mais `self.db.bind.dialect.name`
   - Substituído por detecção segura via `sqlalchemy.inspect(self.db.get_bind()).dialect.name` ou equivalente compatível com SQLAlchemy 2.x
   - Sistema não emite `SADeprecationWarning` ao iniciar

2. **AC2 — Consolidação dos adaptadores SQL**
   - Existe apenas **1** módulo adaptador canônico: `app/core/sql_adapter.py`
   - `conf/sql_adapter.py` e `app/utils/sql_adapters.py` são removidos ou reexportam do canônico
   - Todos os repositórios que usavam os adaptadores antigos importam do canônico
   - Nenhuma funcionalidade é perdida (todos os métodos do adapter antigo existem no canônico)

3. **AC3 — `db_helpers._adapt_sql()` funcional**
   - A função `_adapt_sql()` em `app/core/db_helpers.py` passa o SQL pelo adaptador real (não é mais no-op)
   - Qualquer raw SQL com funções MySQL passado por `exec_sql()` / `fetch_one()` / `fetch_all()` é corretamente adaptado para SQLite

4. **AC4 — Script `export_seed.py`**
   - Existe script `scripts/export_seed.py` que:
     - Conecta ao MySQL (via `.env` ativo)
     - Exporta schema + dados de referência (tabelas: `bandeiras`, `formas_pagamento`, `taxas`, `contextos`, `termos_filtraveis`, `usuarios`) para `data/concilie.db`
     - Gera log de quantidades exportadas por tabela
     - Emite erro claro se tabelas de processamento (vendas, recebiveis) forem incluídas acidentalmente
   - Script é idempotente (pode ser rodado múltiplas vezes, sempre gera seed limpo)

5. **AC5 — Smoke test SQLite**
   - Existe `tests/test_sqlite_smoke.py` que:
     - Inicializa engine SQLite em banco em memória (`:memory:`)
     - Cria todas as tabelas via `Base.metadata.create_all()`
     - Executa as queries críticas de cada repository (bandeiras, taxas, contextos, period grouping, calculo preview)
     - Verifica que **nenhuma query levanta exceção** no SQLite
   - Todos os testes do smoke passam: `pytest tests/test_sqlite_smoke.py -v`

6. **AC6 — Documentação do processo de build**
   - `docs/GUIA_DISTRIBUICAO.md` atualizado com seção "Build e Distribuição" contendo:
     - Pré-requisitos para gerar distribuição
     - Comando para rodar `export_seed.py`
     - Comando para build PyInstaller (referenciando `dist_main.py`)
     - Como verificar integridade do `.exe` gerado
     - Como trocar entre MySQL e SQLite localmente (via `.env`)

## Dev Notes

### Contexto do Problema (originado no @analyst — Brownfield Discovery 2026-03-19)

O sistema Financial distribui o binário completo (PyInstaller) para clientes com SQLite. O backend FastAPI deve funcionar **identicamente** em MySQL (dev/servidor) e SQLite (distribuição). A infra de adaptação existe mas tem gaps:

### Arquivos-chave

| Arquivo | Papel | Situação |
|---------|-------|----------|
| `apps/api/app/core/database.py` | Engine switching por `DATABASE_TYPE` | ✅ OK |
| `apps/api/app/core/sql_adapter.py` | Adaptador canônico (DATE_FORMAT↔strftime, CONCAT↔\|\|, etc.) | ✅ OK — torna-se o único |
| `conf/sql_adapter.py` | Adaptador legado | ⚠️ Duplicado — deprecar/reexportar |
| `apps/api/app/utils/sql_adapters.py` | Terceiro adaptador | ⚠️ Duplicado — deprecar/reexportar |
| `apps/api/app/core/db_helpers.py` | Helpers DB — `_adapt_sql()` é no-op | 🔴 Corrigir |
| `apps/api/app/repositories/analista_repository.py:16` | `self.db.bind.dialect.name` | ⚠️ Depreciado SQLAlchemy 2.x |
| `apps/api/app/repositories/calculo_repository.py:11` | `self.db.bind.dialect.name` | ⚠️ Depreciado SQLAlchemy 2.x |
| `apps/api/app/dist_main.py` | Entry point de distribuição, seed SQLite | ✅ OK |
| `apps/api/scripts/export_seed.py` | Script de export MySQL→SQLite | 🔴 Não existe — criar |

### Como detectar dialeto corretamente (SQLAlchemy 2.x)

```python
# ERRADO (depreciado):
self.dialect = self.db.bind.dialect.name

# CORRETO (SQLAlchemy 2.x compatível):
from sqlalchemy import inspect
self.dialect = inspect(self.db.get_bind()).dialect.name

# Ou via engine diretamente se disponível:
from app.core.database import engine
self.dialect = engine.dialect.name
```

### Distribuição — fluxo atual (dist_main.py)

```
PyInstaller bundle
    ├── data/concilie.db  ← seed gerado por export_seed.py
    └── web_dist/         ← Next.js static export (pnpm build)

Na primeira execução pelo cliente:
    dist_main.py → copia data/concilie.db → %APPDATA%\Financial\financial.db
    FastAPI inicia com DATABASE_TYPE=sqlite apontando para o AppData
```
