# Story 3.45 — Pré-processamento de Relatório com Parquet e Emissão por Modelo

## Status: Ready for Review

## Story

**Como** usuário do sistema,
**quero** pré-processar os dados de um processamento uma única vez e depois emitir qualquer combinação de modelos de relatório (analítico, sem capa, sintético, Excel) marcando checkboxes,
**para que** a geração de relatórios seja rápida e não reprocesse o banco a cada emissão.

---

## Context

Atualmente, cada geração de relatório executa todas as queries MySQL, cálculos Polars e montagem de DataFrames do zero. Isso é lento e desperdiçado quando o usuário quer apenas trocar o modelo (ex: gerar sem capa depois de já ter gerado analítico).

A nova arquitetura separa em duas etapas:
1. **Pré-processamento**: roda uma vez, salva cada seção como `.parquet` em `parquet_cache/{processamento_id}/`
2. **Emissão**: lê os parquets necessários para o(s) modelo(s) selecionado(s) e renderiza o template

Os modelos de relatório são cadastrados no banco (`modelos_relatorio`) com referência ao arquivo de template e às seções necessárias. O template Excel usa `templates/template_excel.xml` (Jinja2 + SpreadsheetML), igual aos HTMLs.

Parquet é invalidado apenas por correção ou recálculo explícito — sem TTL automático. A rotina de update não estará no deploy.

---

## Scope

**IN:**
- Pré-processamento de dados para Parquet por seção
- Tabela `modelos_relatorio` no banco com CRUD básico
- Emissão por checkbox de modelos ativos
- Template Excel via Jinja2 + SpreadsheetML
- Invalidação de cache por correção/recálculo

**OUT:**
- Agendamento automático de pré-processamento (sem scheduler no deploy)
- Migração dos relatórios mensais (relatório mensal continua fluxo atual)
- Gráficos ou pivot no Excel
- Notificações de progresso em tempo real (WebSocket)

## Technical Notes

### Estrutura de parquets
```
parquet_cache/
  {processamento_id}/
    vendas_calculos.parquet
    perdas_semestre.parquet
    taxas_minmax.parquet
    contagem_transacoes.parquet
    recebiveis_sumario.parquet
    tabela_consolidada.parquet
    evidencias.parquet
    dados_bancarios.parquet
    _meta.json   ← { gerado_em: ISO, calc_tipo: str }
```

### Tabela modelos_relatorio (seed inicial)
```sql
INSERT INTO modelos_relatorio (nome, template_arquivo, tipo, secoes_necessarias, ativo) VALUES
('Analítico',        'template_relatorio.html',    'html', '["vendas_calculos","perdas_semestre","taxas_minmax","contagem_transacoes","recebiveis_sumario","dados_bancarios","evidencias"]', 1),
('Analítico Sem Capa','template_sem_capa.html',    'html', '["vendas_calculos","perdas_semestre","taxas_minmax","contagem_transacoes","recebiveis_sumario","dados_bancarios","evidencias"]', 1),
('Sintético',        'template_relatorio_sintetico.html','html','["perdas_semestre","contagem_transacoes"]', 1),
('Excel',            'template_excel.xml',          'xml',  '["vendas_calculos","perdas_semestre","taxas_minmax","contagem_transacoes","recebiveis_sumario","dados_bancarios"]', 1);
```

### Jinja2 Excel — contexto esperado
```python
contexto = {
    "abas": [
        { "nome": "1. Vendas Completas", "colunas": [...], "linhas": [[...], ...] },
        ...
    ]
}
env.tests["number"] = lambda v: isinstance(v, (int, float))
```

### Invalidação de parquet
- `analise-correcoes`: após salvar correção → chamar `invalidar_parquet(processamento_id)`
- Recálculo: após confirmar recálculo → chamar `invalidar_parquet(processamento_id)`
- Sem TTL, sem scheduler

### ⚠️ Regra crítica de implementação — Fluxo paralelo (NÃO tocar no fluxo atual)

`gerar_relatorio_html()` e `gerar_relatorio_mensal_html()` em `modules/reports.py` **NÃO devem ser modificadas**.

As funções de cálculo já existem como funções independentes em `modules/reports.py`:
```
calcular_perdas_por_semestre()
calcular_min_max_taxas_agrupado()
calcular_contagem_taxas_agrupado()
calcular_sumario_recebiveis()
calcular_tabela_consolidada_mensal()
obter_dados_bancarios_distintos()
obter_evidencias_transacoes()
```

`preprocessamento_service.py` deve **importar e reutilizar** essas funções — nunca reescrevê-las.

O endpoint antigo `/relatorios/gerar` deve continuar funcionando identicamente durante toda a implementação. O frontend só migra para os novos endpoints após validação end-to-end do novo fluxo.

```
FLUXO ATUAL (intocado)        FLUXO NOVO (paralelo)
gerar_relatorio_html()        preprocessar_relatorio()
  → calcular_perdas()    ←──────→ calcular_perdas()
  → calcular_taxas()     ←──────→ calcular_taxas()
  → renderiza HTML               → salva .parquet

                                 emitir_modelo()
                                   → lê .parquet
                                   → renderiza template
```
