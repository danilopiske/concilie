## Story

**As a** operador do Concilie,
**I want** gerar automaticamente uma carta de contestação formal com base nos desvios detectados (taxa contratada vs cobrada), acompanhar o status da contestação e calcular o valor total a recuperar por cliente,
**so that** eu possa transformar a análise de abusividade em ação concreta de recuperação financeira com documentação rastreável.

## Contexto Técnico

- **Dependência direta de Story 3.11** — `TaxaContratada` e `DesvioTaxa` devem existir antes desta story
- `AbusividadeRelatorioService.gerar_html()` — padrão de geração de HTML a reutilizar
- `RelatorioEditor.tsx` — editor TipTap reutilizável para editar a carta
- `relatorio_service.save_edit()` — padrão de persistência de HTML
- `apps/web/src/app/(dashboard)/relatorios/editor/page.tsx` — padrão de editor a replicar
- Templates de carta: documentos jurídicos com merge fields (cliente, adquirente, período, valores)

## Acceptance Criteria

1. **AC1 — Model `Contestacao`**
   - Nova tabela `contestacoes`:
     - `id` (UUID, PK)
     - `cliente_id` (Integer, FK → clientes.cliente_id)
     - `processamento_id` (Integer, FK → processamentos, nullable)
     - `adquirente` (String 100)
     - `periodo_inicio` (Date)
     - `periodo_fim` (Date)
     - `valor_excesso_total` (Float) — soma dos `valor_excesso_estimado` dos desvios
     - `status` (String 30): `rascunho` | `enviada` | `em_analise` | `deferida` | `indeferida`
     - `html_carta` (Text, nullable) — HTML editado da carta
     - `created_by` (String 100)
     - `created_at` (DateTime)
     - `updated_at` (DateTime)

2. **AC2 — Geração automática da carta de contestação**
   - `POST /contestacoes/gerar` (body: `cliente_id`, `processamento_id`)
   - Busca `DesvioTaxa` da comparação (Story 3.11) para o processamento
   - Renderiza template Jinja2 `template_contestacao.html` com merge fields:
     - `{{cliente_nome}}`, `{{adquirente}}`, `{{periodo}}`, `{{valor_total_excesso}}`
     - Tabela de desvios por bandeira/modalidade
   - Salva em `contestacoes/{uuid}.html`, cria registro `Contestacao` com `status=rascunho`
   - Retorna `{"contestacao_id": "uuid"}`

3. **AC3 — CRUD e endpoints de contestação**
   - `GET /contestacoes?cliente_id={id}` — lista contestações do cliente com status
   - `GET /contestacoes/{id}` — detalhe + HTML
   - `PUT /contestacoes/{id}/status` — atualiza status (body: `{"status": "enviada"}`)
   - `POST /contestacoes/{id}/save-edit` — salva HTML editado (mesmo padrão de `relatorio_service.save_edit`)
   - `GET /contestacoes/{id}/download` — `FileResponse` do HTML
   - `DELETE /contestacoes/{id}` — remove (somente `status=rascunho`)

4. **AC4 — Template Jinja2 da carta**
   - Criar `relatorios/template_contestacao.html` com estrutura formal:
     - Cabeçalho: dados do cliente, data, destinatário (adquirente)
     - Parágrafo de identificação e objeto da contestação
     - Tabela de desvios por bandeira/modalidade (taxa contratada, cobrada, desvio, valor excesso)
     - Cálculo total do valor a recuperar
     - Seção `<section class="editor-appendix"></section>` para edição livre
     - Rodapé com campos de assinatura
   - Estilo profissional (CSS inline, imprimível)

5. **AC5 — Página `/contestacoes` no frontend**
   - Lista todas contestações com filtro por cliente e status
   - Badges: 📝 rascunho / 📤 enviada / 🔄 em análise / ✅ deferida / ❌ indeferida
   - Botão "Nova Contestação" por cliente → seleciona processamento → chama `POST /contestacoes/gerar`
   - Botão "Editar" → abre `RelatorioEditor` com o HTML da carta
   - Botão "Atualizar Status" → dropdown inline
   - KPI no topo: "Total a Recuperar" (soma `valor_excesso_total` de contestações não indeferidas)
   - Link "Contestações" no sidebar (seção nova ou em "Análise e Correções")

6. **AC6 — Qualidade**
   - `ruff check app/` → 0 erros
   - `pytest tests/ -v` → todos passando
   - `contestacoes/` adicionado ao `.gitignore`

### Bug Fixado Durante Review

- **`_TEMPLATES_DIR` — path errado (blocker corrigido)**: `parent * 4` apontava para `apps/relatorios` em vez de `apps/api/relatorios`, causaria `TemplateNotFound` em todo `POST /gerar`. Corrigido para `parent * 3`. `import os` não utilizado também removido.

### Concerns (não-blockers)

1. **`created_by` hardcoded como "sistema"** — deveria usar o usuário autenticado do token JWT; aceitável para MVP
2. **Sem testes unitários para `contestacao_service`** — lógica de geração e CRUD sem cobertura automatizada
3. **`RelatorioEditor` importado via `m.RelatorioEditor || m.default`** — acoplamento frágil ao export do componente; depende do ambiente de runtime
4. **Browser check (5.4) pendente** — fluxo gerar → editar → salvar → download não validado visualmente

### Decision
**PASS with CONCERNS** — Bug de path corrigido inline. Lógica core correta, segurança adequada (DELETE apenas rascunho, parameterizado). Pronto para merge.
