## Story

**As a** usuário do sistema Financial,
**I want** poder editar o relatório gerado em uma interface de editor de texto online, inserindo seções predefinidas via comando `/` (slash commands),
**so that** eu possa personalizar e enriquecer o relatório antes de exportá-lo, sem precisar editar o HTML bruto, e manter uma biblioteca de seções reutilizáveis cadastradas.

## Acceptance Criteria

1. **AC1 — Página de Editor de Relatório**
   - Existe rota `/relatorios/editor` no Next.js
   - A página aceita `?task_id=<uuid>` como query param e carrega o HTML do relatório gerado
   - O conteúdo do relatório é renderizado dentro de um editor rich-text (TipTap)
   - O editor suporta formatação básica: negrito, itálico, headings (H1–H3), listas, parágrafos
   - Botão "Salvar" persiste o conteúdo editado via API
   - Botão "Exportar HTML" gera download do arquivo editado

2. **AC2 — Slash Commands no Editor**
   - Ao digitar `/` em qualquer linha vazia do editor, aparece um menu flutuante de sugestões
   - O menu lista as tags cadastradas (nome + descrição curta), filtráveis por digitação após `/`
   - Selecionar uma tag insere seu `conteudo_padrao` no ponto do cursor como um bloco de conteúdo
   - O menu fecha ao pressionar `Escape` ou clicar fora
   - Navegação no menu via teclado (`↑`, `↓`, `Enter`)
   - Máximo de 10 itens visíveis no menu (scroll se houver mais)

3. **AC3 — Tela de Gestão de Tags**
   - Existe rota `/relatorios/tags` no Next.js
   - A tela lista todas as tags cadastradas em tabela com colunas: Nome, Tipo, Descrição, Ações
   - Formulário de criação de tag com campos:
     - `nome`: string, obrigatório, único, máx 50 chars
     - `tipo`: enum (`secao`, `clausula`, `assinatura`, `cabecalho`, `rodape`), obrigatório
     - `descricao`: string opcional, máx 200 chars
     - `conteudo_padrao`: textarea HTML/Markdown, obrigatório
     - `ativo`: boolean, default `true`
   - Ações: editar inline ou modal, excluir com confirmação
   - Tags inativas aparecem acinzentadas e não aparecem no menu slash do editor
   - Validação client-side antes de submeter

4. **AC4 — Backend: API de Tags (`/api/v1/relatorio-tags`)**
   - `GET /` — listar todas as tags (param: `ativo=true|false|all`, default `true`)
   - `POST /` — criar nova tag (valida unicidade de `nome`)
   - `GET /{tag_id}` — retornar tag por ID
   - `PUT /{tag_id}` — atualizar tag
   - `DELETE /{tag_id}` — excluir tag (soft delete: seta `ativo=false`) ou hard delete se nunca usada
   - Todos os endpoints requerem autenticação (`get_current_user`)
   - Retornar HTTP 422 com mensagem clara para `nome` duplicado

5. **AC5 — Persistência do Relatório Editado**
   - `POST /api/v1/relatorios/tasks/{task_id}/save-edit` — salva conteúdo editado
   - Salva o HTML editado em arquivo `{task_id}_edited.html` no diretório `relatorios_gerados/`
   - Atualiza `relatorio_tasks.result_path` para apontar ao arquivo editado
   - Botão de download em `/relatorios/editor` baixa a versão mais recente (editada ou original)

6. **AC6 — Integração na Navegação**
   - Link "Editar" disponível no histórico de relatórios (`RelatorioHistory.tsx`) para tasks com status `SUCCESS`
   - Link "Gerenciar Tags" no menu lateral ou cabeçalho da seção de relatórios
   - Breadcrumb: `Relatórios > Editor` e `Relatórios > Tags`

## Dev Notes

### Contexto do Módulo Atual

O módulo de relatórios (`apps/api/app/api/v1/endpoints/relatorios.py`) já possui endpoints assíncronos de geração, histórico e download. A tabela `relatorio_tasks` armazena o caminho do HTML gerado em `result_path`. O frontend em `apps/web/src/app/(dashboard)/relatorios/page.tsx` e `RelatorioHistory.tsx` exibem histórico e botões de download.

**Não existe** atualmente nenhuma funcionalidade de edição de relatório nem sistema de tags inseríveis. A tabela `termos_filtraveis` (existente) é para filtros de vendas, **não** para tags do editor — são conceitos distintos.

### Decisão de Biblioteca: TipTap

TipTap é o editor rich-text recomendado para Next.js/React. Suporte nativo a extensões de slash commands. Alternativas consideradas: Slate.js (mais complexo), Quill (menos mantido), ProseMirror direto (muito baixo nível).

- Pacotes base: `@tiptap/react @tiptap/pm @tiptap/starter-kit`
- Para slash commands: verificar `tiptap-extension-slash-commands` ou implementar via `@tiptap/suggestion`

**Verificar compatibilidade** com versão do Next.js e React do projeto antes de instalar.

### Schema da Tabela `relatorio_tags`

```python
class RelatorioTag(Base):
    __tablename__ = "relatorio_tags"
    id = Column(Integer, primary_key=True, autoincrement=True)
    nome = Column(String(50), unique=True, nullable=False, index=True)
    tipo = Column(String(20), nullable=False)  # secao|clausula|assinatura|cabecalho|rodape
    descricao = Column(String(200), nullable=True)
    conteudo_padrao = Column(Text, nullable=False)
    ativo = Column(Boolean, default=True, nullable=False)
    created_at = Column(DateTime, default=datetime.utcnow)
    updated_at = Column(DateTime, default=datetime.utcnow, onupdate=datetime.utcnow)
```

### Fluxo de Slash Commands

```
Usuário digita "/" no editor
    ↓
TipTap Suggestion extension intercepta
    ↓
Frontend faz GET /api/v1/relatorio-tags?ativo=true
    ↓
Menu flutuante exibe lista filtrada pelo texto após "/"
    ↓
Usuário seleciona tag (Enter ou clique)
    ↓
Editor insere conteudo_padrao da tag no ponto do cursor
    ↓
Texto "/" é removido (substituído pelo conteúdo)
```

### Endpoint Save Edit

```
POST /api/v1/relatorios/tasks/{task_id}/save-edit
Body: { "html_content": "<html>...</html>" }

1. Valida que task existe e está com status SUCCESS
2. Salva em: relatorios_gerados/{task_id}_edited.html
3. UPDATE relatorio_tasks SET result_path = novo_caminho WHERE id = task_id
4. Return: { "success": true, "path": "..." }
```
