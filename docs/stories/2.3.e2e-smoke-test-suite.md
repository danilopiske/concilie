## Story

**As a** desenvolvedor/mantenedor do sistema Concilie,
**I want** ter uma suite de smoke tests E2E com Playwright cobrindo os fluxos críticos,
**so that** regressões em funcionalidades-chave sejam detectadas automaticamente antes de qualquer deploy.

## Acceptance Criteria

1. **AC1 — Setup do Playwright**
   - Playwright instalado e configurado em `apps/web/` (ou raiz do monorepo)
   - Arquivo `playwright.config.ts` com configuração de base URL, timeouts e reporters
   - Script `npm run test:e2e` executa os testes E2E
   - Testes rodam contra servidor local (`npm run dev` ou `npm run start`)

2. **AC2 — Smoke test: Autenticação**
   - Teste: login com credenciais válidas → redireciona para dashboard
   - Teste: login com credenciais inválidas → exibe mensagem de erro
   - Teste: acesso à rota protegida sem login → redireciona para `/login`

3. **AC3 — Smoke test: Módulo de Relatórios**
   - Teste: página `/relatorios` carrega com histórico visível (ou estado vazio)
   - Teste: botão de gerar relatório inicia task e exibe status em polling
   - Teste: link "Editar" em task com status SUCCESS abre `/relatorios/editor?task_id=...`

4. **AC4 — Smoke test: Editor de Relatório**
   - Teste: página `/relatorios/editor?task_id=<valid_id>` carrega iframe com relatório
   - Teste: digitar texto no TipTap editor → texto aparece na área de edição
   - Teste: digitar `/` → menu de slash commands aparece
   - Teste: botão "Salvar" com conteúdo → exibe "✓ Relatório salvo com sucesso!"

5. **AC5 — Smoke test: Gestão de Tags**
   - Teste: página `/relatorios/tags` carrega lista de tags (ou estado vazio)
   - Teste: criar nova tag com campos válidos → tag aparece na lista
   - Teste: excluir tag → tag removida da lista após confirmação

6. **AC6 — Integração com CI**
   - Testes E2E configurados no GitHub Actions workflow (`.github/workflows/`)
   - Job `e2e` roda após `build` com sucesso
   - Relatório HTML de resultados como artifact do CI
   - Falha em qualquer smoke test bloqueia o merge da PR

## Dev Notes

### Contexto

O projeto já tem CI com pytest (Sprint 3). Os smoke tests E2E complementam os testes unitários/integração do backend com validação de fluxo real no frontend.

### Estrutura esperada

```
apps/web/
├── playwright.config.ts
├── e2e/
│   ├── auth.spec.ts
│   ├── relatorios.spec.ts
│   ├── editor.spec.ts
│   └── tags.spec.ts
└── package.json  # + script test:e2e
```

### Estratégia de fixtures

- Usar `page.route()` para mockar chamadas de API onde necessário (isolamento)
- Para testes que exigem dados reais (editor com task_id), usar seed de dados ou criar via API no `beforeEach`
- Credenciais via `process.env.E2E_USER` / `process.env.E2E_PASSWORD`

### Configuração base sugerida

```typescript
// playwright.config.ts
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './e2e',
  baseURL: process.env.PLAYWRIGHT_BASE_URL ?? 'http://localhost:3000',
  timeout: 30000,
  reporter: [['html', { outputFolder: 'playwright-report' }]],
  use: {
    headless: true,
    screenshot: 'only-on-failure',
  },
});
```

### Comandos úteis

```bash
cd apps/web
npm run test:e2e                    # Rodar todos os testes
npx playwright test --headed        # Com browser visível
npx playwright test e2e/auth.spec.ts  # Rodar spec específico
npx playwright show-report          # Ver relatório HTML
```
