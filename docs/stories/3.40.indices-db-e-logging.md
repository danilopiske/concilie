# Story 3.40 — Índices de Performance e Logging de Abusividade

## Status: Done

## Contexto

Análise de performance identificou duas melhorias críticas:
1. Queries de `vendas_calculos` sem índices compostos — full table scan em tabelas com milhões de linhas
2. O serviço `abusividade_relatorio_service.py` não registra timing, volume processado nem falhas parciais, dificultando diagnóstico de lentidão

---

## Story

**Como** administrador do sistema,
**Quero** que queries de vendas respondam rápido e que o processamento de abusividade seja rastreável,
**Para** garantir SLA aceitável em relatórios e facilitar diagnóstico de problemas de performance.

---

## Acceptance Criteria

### AC1 — Índices compostos em `vendas_calculos`
- [x] Índice composto criado: `(calc_id, data_venda)`
- [x] Índice composto criado: `(bandeira, forma_pagamento)`
- [x] Índice simples criado: `(cliente_id)` (se ainda não existir)
- [x] Migração Alembic criada e aplicada (0001_add_indexes_vendas_calculos.py)
- [ ] `EXPLAIN` nas queries de abusividade e comparativo confirma uso dos índices *(pós-deploy com DB real)*

### AC2 — Logging no serviço de abusividade
- [x] Log no início do processamento: `"Iniciando cálculo abusividade cliente={id} período={inicio}..{fim}"`
- [x] Log no fim: `"Concluído cliente={id} linhas={n} tempo={elapsed:.2f}s"`
- [x] Log para cada lote processado (se houver paginação): `"Lote {n} processado: {count} registros"`
- [x] Exceções capturadas com `logger.exception(...)` — stack trace completo no log
- [x] Nível de log: `INFO` para início/fim, `DEBUG` para lotes, `ERROR` para exceções

### AC3 — Índice em `relatorio_tasks`
- [x] Índice composto criado: `(cliente_id, status, created_at DESC)`
- [x] Melhora query de listagem de relatórios pendentes

### AC4 — Testes
- [ ] Teste confirma que migração aplica e reverte sem erro *(pós-deploy com DB real)*
- [x] Teste de unidade para logging: verifica que `logger.info` é chamado com os campos esperados
