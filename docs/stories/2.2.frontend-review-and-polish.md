## Story

**As a** desenvolvedor/mantenedor do sistema Concilie,
**I want** ter o frontend Next.js revisado e polido,
**so that** todas as páginas estejam consistentes, sem erros de console, com UX adequada e prontas para uso em produção.

## Acceptance Criteria

1. **AC1 — Consistência visual e de componentes**
   - Todas as páginas usam os componentes `Button`, `Panel`, `Input` de `@/components/ui/`
   - Não há estilos inline redundantes que conflitem com Tailwind
   - Paleta de cores consistente via variáveis CSS (`--primary`, `--secondary`)
   - Todas as páginas têm breadcrumb ou indicação de localização no app

2. **AC2 — Tratamento de estados de carregamento e erro**
   - Todas as páginas com fetch assíncrono mostram `Loader2` durante carregamento
   - Erros de API são capturados e exibidos em banner visível (não silenciados)
   - Estados vazios (listas sem dados) têm mensagem descritiva + CTA quando aplicável
   - Não há `console.error` sem feedback visual correspondente ao usuário

3. **AC3 — Formulários e validação**
   - Todos os formulários têm validação client-side antes do submit
   - Mensagens de erro de campo são exibidas inline (não apenas toast/alert)
   - Campos obrigatórios estão marcados visualmente
   - Submit desabilitado enquanto loading/saving está ativo

4. **AC4 — Navegação e roteamento**
   - Todas as rotas do Next.js estão funcionando (sem 404 inesperados)
   - Links de navegação no sidebar/header refletem a rota ativa
   - `useSearchParams()` está dentro de `<Suspense>` (Next.js 14+ requirement)
   - Redirecionamentos de auth funcionam corretamente (não logado → login)

5. **AC5 — Performance básica**
   - Nenhum componente importa biblioteca pesada no bundle principal sem lazy loading
   - Imagens (se houver) usam `next/image` com dimensões definidas
   - Sem loops `useEffect` infinitos (dependency arrays corretos)
   - `RelatorioEditor.tsx`: TipTap extension de slash commands não recria a cada render

6. **AC6 — TypeScript e linting**
   - `npm run typecheck` passa sem erros em `apps/web/`
   - `npm run lint` passa sem erros em `apps/web/`
   - Sem `@ts-ignore` ou `any` desnecessários em componentes principais
   - Props interfaces definidas para todos os componentes customizados

## Dev Notes

### Contexto

O frontend usa Next.js 14+ com App Router. Componentes UI customizados em `apps/web/src/components/ui/`. As páginas principais a revisar são:

```
apps/web/src/app/(dashboard)/
├── relatorios/
│   ├── page.tsx                    # Listagem e geração
│   ├── RelatorioHistory.tsx        # Histórico de tasks
│   ├── editor/page.tsx             # Editor TipTap + iframe
│   └── tags/page.tsx               # Gestão de tags
├── gestao/
│   ├── clientes/
│   ├── ecs/
│   ├── bandeiras/
│   └── taxas/
├── conciliacao/
├── importacao/
└── calculos/
```

### Comandos úteis

```bash
cd apps/web
npm run typecheck
npm run lint
npm run build  # Para detectar erros de build
```

### Observações conhecidas

- `RelatorioEditor.tsx`: `buildSlashExtension(tags)` é chamado em `useRef` (correto para inicialização) mas também no `useEffect` recria a extensão — o editor TipTap não pega a nova instância sem destruir/recriar. Isso pode ser melhorado ou documentado.
- `editor/page.tsx`: Usa `useSearchParams()` diretamente — verificar se está dentro de `<Suspense>`.
